Skill 作成は「ディレクトリ → SKILL.md → 再起動」の3ステップ

Step 1

ディレクトリを作成

  • 配置先：~/.claude/skills/<skill-name>/
  • ディレクトリ名は name と一致させる
  • 例：mkdir -p ~/.claude/skills/pr-description

Step 2

SKILL.md を作成

  • frontmatter に name と description を記述
  • --- 以降に Claude への指示書本文
  • description は「Use when …」を含めると誤発火が減る

Step 3

Claude Code を再起動

  • Skill は 起動時にスキャンされる
  • 利用可能 Skill 一覧で名前を確認
  • 自然文で依頼してロードされるか検証

SKILL.md は「frontmatter = 看板」「本文 = 中身」の二層構造

name は識別子，description がマッチングキー，本文が指示書

  • frontmatter は --- で囲み，name（kebab-case1）と description（自然文）を必ず書く
  • frontmatter 以下は Claude が Skill を発火したときに読み込む指示書
  • description は具体的に書くほど 意図したタイミングだけで起動するようになる

最小構造の例

---
name: pr-description
description: Writes pull request descriptions.
  Use when creating a PR, writing a PR, or when
  the user asks to summarize changes for a pull
  request.
---

When writing a PR description:

1. Run `git diff main...HEAD` to see all changes
2. Write a description following this format:

## What
One sentence explaining what this PR does.

## Why
Brief context on why this change is needed.

## Changes
- Bullet points of specific changes made

フィールドの役割

フィールド 役割 書き方のコツ
name 識別子 kebab-case，ディレクトリ名と一致
description マッチング “Use when …” を含め具体的に
本文 指示書 手順・出力テンプレを明示

コツ

  • frontmatter を変えたら Claude Code を再起動しないと反映されない
  • 本文はマッチ後に遅延ロード．長くてもコンテキストを圧迫しない

  1. 最大64文字．小文字，数字，ハイフンのみ使用可能．先頭．末尾ハイフン禁止 ^[a-z0-9]+(-[a-z0-9]+)*$

「Skill 一覧の質問」と「ターミナル出力」で登録を目視確認

作成直後はまず一覧で名前が出るかを見る．これで frontmatter 構文ミスを早期検知できる

① 自然文で問い合わせる

  • What skills are available? と聞くだけ
  • スラッシュコマンドではなく 自然文で OK

② 一覧出力を確認

  • Skill: <name> と Description: ... が並ぶ
  • 期待した pr-description が含まれているかを目視
  • 出ていなければ frontmatter 構文・ディレクトリ名を疑う

指示に応じて Skill が自動マッチされロードログが可視化される

Skill 名を呼ばずとも．description の意味マッチで Claude が選定し，ログ表示される

起動の流れ

  1. ユーザーが 自然文で依頼を入力
    • 例：Write a PR description for my changes.
  2. Claude Code が登録 Skill の description と意味マッチ
  3. マッチした Skill 名と説明が IDE 上に表示される
  4. 確認後，本文がコンテキストに展開され実行される

ここがデバッグの勘所

  • 表示が出れば description が効いている証拠
  • 出ない・別 Skill が出る → description の文言を見直す

Claude Code は4ヶ所をスキャンし「name + description」だけを先読みする

起動時：軽量メタロード

  • スキャン対象は4ヶ所
    • managed-settings.json（Enterprise）
    • ~/.claude/skills/（Personal）
    • <repo>/.claude/skills/（Project）
    • <repo>/.claude-plugin/plugin.json（Plugins）
  • 各 SKILL.md から name と description のみを読む
  • 本文はこの段階では読み込まない

リクエスト時：意味マッチ + 確認

  • ユーザー依頼を 全 description と意味的に照合
  • 例：「この関数を説明して」と「explain code with diagrams」が一致
  • マッチした Skill はロード前に 確認プロンプトが出る
  • 承認後に本文が読み込まれ，指示書に従って実行

REMARKS

  • メタ情報だけ先読みする設計のおかげで，何十個 Skill を置いてもコンテキストを圧迫しない
  • 確認プロンプトがあるので「どの Skill が動いたか」を毎回ユーザーが把握できる

同名 Skill 衝突は，Enterprise → Personal → Project → Plugins の順で優先される

順位 種別 配置場所 役割
1st Enterprise managed-settings.json 組織が強制する標準（全社統一のレビュー基準など）
2nd Personal ~/.claude/skills/ 個人の作法を全プロジェクトに適用
3rd Project <repo>/.claude/skills/ リポジトリ同梱でチームに配布
4th Plugins <repo>/.claude-plugin/plugin.json インストールしたプラグイン由来

衝突を避ける命名のコツ

  • 汎用的な名前（例：review）は避ける
  • 用途を含めて命名：frontend-review，backend-review
  • チーム独自なら <team>-<task> の prefix も有効

この順序の意図

  • Enterprise が最優先：組織の合意がローカル設定に勝つ思想
  • Plugins が最下位：個人・プロジェクト側で上書き可能な拡張ポイント
  • 衝突時の解は 命名整理が原則

SKILL.md 更新・削除は，どちらもセッション再起動が必須

ファイル操作はシンプルだが「再起動忘れ」が最頻出のハマりポイント

  • 更新したいとき：SKILL.md を編集して Claude Code を再起動
  • 削除したいとき：Skill ディレクトリを丸ごと削除して 再起動
  • description だけの変更でも再起動が必要．起動時にしかメタを読み直さないため

よくある事故

  • 編集したのに古い挙動 → 再起動漏れ
  • 起動しなくなった → ディレクトリ名と name の不一致
  • 想定外に発火する → description が抽象的すぎ

改善サイクル

  • 誤発火・空振りを観察し description を磨く
  • 本文は試行を重ねながら手順を具体化
  • 安定したら個人 → プロジェクトへ昇格も検討

良い description は「何をする」と「いつ使う」を両方含む

漠然とした description は誤発火・空振りの元．具体的なキーワードと発動条件をセットで書く

  • description は Claude がマッチングに使う唯一の手がかり．本文は読まれない
  • What（何をする Skill か）と When（いつ使うか）を両方書く．Use when ... は便利なテンプレ
  • 発火しないときはユーザーが実際に使う言い回しを description に追記する

弱い description（誤発火しやすい）

  • Helps with docs.
    • 何を「help」するかが不明瞭
    • 「docs」もカバー範囲が広すぎる
  • 結果：意図しない依頼でも発火しがち

強い description（誤発火しにくい）

  • Writes pull request descriptions. Use when creating a PR or summarizing changes for review.
    • What：「PR description を書く」
    • When：「PR を作るとき」「変更を要約するとき」

Skill は「成果物・タスク・ドメイン」の3観点で切り分ける

「何を作るか」「どう動くか」「どの領域か」のどれを軸にするかで Skill の粒度と再利用性が決まる

成果物単位

  • 「何を作るか」で Skill を切る
  • 最終的に生まれるファイル・ドキュメントのフォーマットを軸にする
  • 例：API仕様書，テスト計画書，運用手順書

タスク単位

  • 「どういう作業をするか」で Skill を切る
  • 手順・判断分岐・典型的な失敗回避を資産化できる
  • 例：パフォーマンステスト実行，DBマイグレーション，セキュリティスキャン

ドメイン単位

  • 「どの分野・領域か」で Skill を切る
  • 専門用語・業務慣習・固有ルール・規制要件を集約
  • Claude の事前知識が古い・不足する領域を埋めるのに有効

複数観点が重なる実タスクは，層に分けた Skill を組み合わせる

一つの巨大 Skill に詰め込むより，観点ごとに分けるほうが再利用性も保守性も上がる

  • 例：「規制対象プロダクトのリリース前チェック」は ドメイン × タスク × 成果物の3層が同時に走る
  • 1つの Skill に詰め込むと 他のプロダクトや別タスクで再利用できなくなる
  • 層ごとに分けておけば ドメインだけ差し替え・タスクだけ流用が可能になる

3層に分解した例：規制対象プロダクトのリリース前チェック

record1:
  層: ドメイン Skill
  Skill の責務:
    - 領域固有の知識・規制を集約
  このケースでの中身:
    - 案件特有の規制要件
    - プロダクト固有ルール

record2:
  層: タスク Skill
  Skill の責務:
    - 作業手順・判断分岐を資産化
  このケースでの中身:
    - セキュリティスキャンの実行手順
    - 失敗パターンと回避手順

record3:
  層: 成果物 Skill
  Skill の責務:
    - 出力フォーマットを統一
  このケースでの中身:
    - 運用手順書としてまとめる
    - 提出用ドキュメントの体裁を整える
