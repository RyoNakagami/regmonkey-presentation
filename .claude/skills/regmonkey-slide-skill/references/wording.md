# 推奨用語変換

技術スライドの信頼性と読みやすさを保つための語彙変換リスト．**絶対遵守ではなく**，各エントリの **Why** と **例外** を読んで文脈に合うときに適用すること．フラットに置換しない．

各エントリは次の構造：

- **元の語** → 推奨置換
  - **Why:** 置換すべき理由（語の由来・印象・聴衆への伝わり方）
  - **例外:** その語をそのまま使ってよい文脈

---

## 1. 慣用・口語的な技術用語の中立化

囲碁将棋・スポーツ・ビジネス慣用句に由来する語は，技術文書では砕けた・曖昧な印象を与える．中立な書き言葉に置き換える．

- **定石** → `一般的`
  - **Why:** 囲碁将棋由来の口語表現．技術スライドでは「業界共通の手法」を曖昧に指してしまい，根拠の弱い断定に見える．`一般的` は頻度，`定型的` はパターン化，`王道` は推奨度を表すのでニュアンスで使い分ける．
  - **例外:** 実際に囲碁・将棋・ゲーム理論・棋譜分析を扱うスライド．

- **肝** → `要点`／`核`／`本質`
  - **Why:** 「肝は〜」は口語的．書き言葉のスライドでは `〜の要点は` のほうが自然．
  - **例外:** なし（基本的に置換推奨）．

- **イケてる** / **ダサい** → `適切`／`不適切`，`望ましい`／`望ましくない`
  - **Why:** 主観的な俗語．技術判断を曖昧にする．
  - **例外:** UI/UX レビューでチームの口語をそのまま引用する場面のみ．

---

## 2. 誇張・煽り表現の事実ベース化

SKILL.md のハウスライティングルールにある「誇張・煽り表現を使わない」の具体マッピング．**抽象的な強調語は，具体的な数字・構造・動詞に置き換える**．

- **驚くほど単純** → `3ステップで済む`／`設定不要で動く`
- **劇的に高速化** → `処理時間が80%減`／`p99 が 200ms → 40ms`
- **圧倒的に強力** → `<具体的な機能>が <具体的な比較対象>より優れている`
- **魔法のように動く** → `自動でロードされる`／`設定なしで認識される`
- **革命的** → 削る（断定できないなら断定しない）
- **完璧** / **究極** → `現状でカバーできる範囲を網羅`／削る
- **秒で** / **あっという間に** / **楽々** → 具体的な所要時間（`5分で`／`1コマンドで`）

**Why（共通）:** 誇張語は技術スライドの信頼性を下げる．聴衆は「で，どれくらい？」と問いを返したくなる．先に数字で答える．

**例外:** 引用・他社プロダクトのキャッチコピーをそのまま示す場合のみ（その場合は引用と明示）．

---

## 3. 書き言葉化（口語 → 文書）

スライド本文（H2 やリスト項目）は口語ではなく書き言葉で書く．

- **〜とか** → `〜など`／`〜や`
- **やっぱり** → `やはり`／削る
- **ちゃんと** → `適切に`／`正しく`
- **すごく** / **めっちゃ** → 具体的な度合い（`大幅に`／数値）
- **〜したり〜したり** → 並列が2項なら `〜や〜を行う`，3項以上なら `・` で列挙
- **〜って** → `〜という`／`〜とは`

**Why:** スライドはプレゼン補助だが文書としても残る．口語は書き起こしで読むと違和感が強い．

**例外:** 講演者ノート（speaker notes）や質疑応答想定の付録スライド．

---

## 4. 製品・ツール文脈での用語統一

スライドが特定の製品（Claude Code・ChatGPT・Cursor 等）の機能を説明する場合，公式ドキュメントの用語に揃える．曖昧な汎用語を使うと，読者が機能の正確な単位を取り違える．

- **会話** → `セッション`（Claude Code 文脈）
  - **Why:** Claude Code の公式用語は "session"．`/clear` で session が始まり，`/exit` まで継続する**明確な単位**．「会話」は雑談的・連続的なニュアンスを含み，どこからどこまでが一つの単位か曖昧になる．特に「CLAUDE.md は◯◯の冒頭に読み込まれる」「Skills は◯◯に追加される」のような **機能の境界条件** を説明するときは `セッション` のほうが技術的に正確で，公式ドキュメントとも整合する．
  - **例外:** ユーザーと AI の対話形式そのものを抽象的に語る場面（「会話形式でやり取りする」のような一般論）．他ツール（ChatGPT・Cline 等）について言及する文脈ではそのツールの公式用語に従う．

**Why（共通）:** 公式用語に揃えると，読者が公式ドキュメント・コミュニティ記事を検索したときに同じキーワードで辿れる．独自の言い換えは技術文書の検索性と互換性を下げる．

---

## 5. 判断ガイド

迷ったら次を自問する：

1. **その語を抜いて意味が変わるか？** 変わらないなら抜く（特に `〜的な`・`まさに`・`いわば`）．
2. **聴衆が「具体的には？」と聞き返したくなるか？** なるなら数字・構造で書き直す．
3. **その語は囲碁将棋・スポーツ・ビジネス慣用句から来ていないか？** 来ているなら中立語に置換できないか検討．
4. **このスライドが文書として配布されたとき違和感があるか？** あるなら書き言葉化する．
5. **特定の製品・ツールの機能を説明しているか？** しているなら公式用語に揃える（`会話` → `セッション` 等）．
